\chapter{Testing}
\label{ch:testing}
% ======================================================================

This chapter provides evidence that GiTrip was tested systematically at unit,
functional, and non-functional levels. The goal is to demonstrate correctness for
core algorithms (merge and scheduler) and reliability for user-facing workflows.

The Project Specification planned three testing categories: unit tests for opening-hours parsing, time arithmetic, merge functions, and CLI command parsing; integration tests for end-to-end auto-plan with mocked routing and merge workflows; and UAT via a survey on Evaluation Day. The final approach deviates in three ways: CLI parsing tests were dropped (C1 was descoped), integration tests use real routing APIs rather than mocked routing, and UAT results are presented in Chapter~\ref{ch:evaluation}.

\subsection{Opening Hours Parsing and Validation}
The opening-hours module tests cover standard weekday ranges (e.g., \texttt{Mo-Fr 09:00-18:00}), wrap-around ranges (e.g., \texttt{Fr-Mo}), the \texttt{off} keyword meaning closed, multi-span days with breaks, compound rules across different days, and empty/null input handling. Validation tests check whether a proposed interval is open, closed, nudged to a next open slot, too long to fit, or exactly fits a boundary.

\subsection{Merge Conflict Detection}
Merge tests cover clean merges where only one side changes, deep-equality behaviour that is independent of key order, deletions (delete-vs-modify yields explicit \texttt{plan-stop-delete} conflict), time conflicts (\texttt{plan-stop-time}), field conflicts (\texttt{plan-stop-field}), whole-plan conflicts when similarity is low (\texttt{plan-whole}), and deterministic ordering of merged days and stops.

\subsection{Planner Time Utilities and Scheduling Logic}
Planner tests cover both time-conversion functions (\texttt{toMin},
\texttt{toHHMM}, including null/empty inputs, negative clamping, and fractional
rounding) and the \texttt{applyOpeningHours} constraint engine, including:
\begin{itemize}
  \item unconstrained scheduling (no constraints; starts at proposed arrival),
  \item clamping to active period,
  \item soft desired windows (\texttt{desiredStart}, \texttt{desiredEnd}),
  \item opening-hours nudging to the next open slot,
  \item hard fixed-time windows (failure when travel arrives too late),
  \item rejection when the stop does not fit within the active window,
  \item rejection when the venue is closed all day.
\end{itemize}

\subsection{Routing Fallback Logic}
Routing tests cover the Haversine fallback estimator and constant-gap defaults used when external routing APIs are unavailable. Tests verify the identity case (same point returns zero travel time), realistic distance estimation (London to Paris within expected range), mode ordering (walking slower than cycling, cycling slower than driving), graceful handling of null or missing coordinates, and correct constant-gap values for each transport mode.

\subsection{External API Connectivity}
API connectivity tests verify that every third-party service required by GiTrip
is correctly configured and reachable at test time:
\begin{itemize}
  \item \textbf{Presence checks} -- each required environment variable
        (\texttt{ORS\_API\_KEY}, \texttt{GOOGLE\_DIRECTIONS\_API\_KEY},
        \texttt{NEWS\_API\_KEY}) is set in \texttt{.env}.
  \item \textbf{ORS (OpenRouteService)} -- a real matrix request and a
        directions request confirm the key is accepted and returns valid
        duration data.
  \item \textbf{Google Directions} -- both driving and transit modes are
        tested; the test asserts that the response status is \texttt{OK}
        (not \texttt{REQUEST\_DENIED}), catching billing or API-enablement
        issues early.
  \item \textbf{GNews} -- a search query confirms the key returns articles
        and a travel-specific query returns relevant results.
  \item \textbf{Open-Meteo} -- both the forecast and archive endpoints are
        tested with a London coordinate, verifying that daily temperature
        and precipitation data are returned in the expected format.
  \item \textbf{Nominatim} -- a geocoding search confirms the endpoint
        returns results with latitude, longitude, and display name. A
        second test verifies that the \texttt{extratags} field (which
        carries opening-hours data from OSM) is present when requested.
\end{itemize}
These tests act as a deployment smoke test: a single failing assertion
pinpoints exactly which external service is misconfigured, rather than
requiring manual investigation of silent routing fallbacks.

\section{Non-functional Testing and Checks}
\subsection{Security Checks}
The security measures from Chapter~\ref{ch:design} (CSP headers, rate limiting, prepared statements, generic error messages) were verified during deployment with input normalisation. These provide baseline hardening but are not a substitute for a full penetration test.

\subsection{Accessibility Checks}
Accessibility was guided by WCAG 2.2. Checks included consistent labelling of controls, avoiding information conveyed only via colour, readable typography and spacing, and basic keyboard navigation for non-map interactions. A more formal audit using automated tools such as Lighthouse is recommended for future work \cite{ref_lighthouse_overview}. The interactive map and timeline
remain difficult to fully validate for accessibility within the project scope.

\subsection{Resilience to External API Failures}
Functional tests included scenarios where routing keys were missing or external
calls failed. In these cases, GiTrip falls back to Haversine-based time estimates
with mode-dependent speed assumptions for ordering, and constant default travel
gaps per mode as a safety margin (as described in Chapter~\ref{ch:design},
Section~5.3). The planner still produces a feasible schedule with reduced
accuracy rather than failing entirely. This was verified by temporarily revoking
API keys and confirming that the auto-scheduler completed without error, the
timeline rendered with approximate travel gaps, and no data corruption occurred
in the repository.

\section{Testing Summary}
The testing evidence demonstrates that the most error-prone components (merge
logic and scheduler constraints) are covered by automated unit tests and that
major user workflows function end-to-end. In terms of success criteria coverage:
planner unit tests and functional scheduling tests support SC2 (scheduling
feasibility), merge unit tests and the merge functional workflow support SC1
(merge usability), and accessibility checks address SC3 (usability and
accessibility). Remaining limitations are largely
usability and accessibility challenges inherent in interactive planning UIs,
rather than basic correctness failures.
