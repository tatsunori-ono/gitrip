\chapter{Conclusions and Future Work}
\label{ch:conclusions}
% ======================================================================

This chapter summarises achievements, critiques the system against its goals,
discusses limitations, and proposes future work.

\section{Conclusions}
GiTrip demonstrates that Git-like version control semantics can be meaningfully
applied to collaborative trip planning. The completed system meets all Must-have
and Should-have objectives from the Project Specification, delivering
repository-based trip storage with commits and branches, constraint-aware
automatic scheduling integrating routing data, interactive timeline and map
editing with commit saving, and three-way merge with a dedicated
conflict-resolution UI.

Evaluation results indicate that users can successfully generate feasible
itineraries (9/10 in UAT Task~T1) and resolve merge conflicts within target
times (9/10 in Task~T2). The System Usability Scale score of 81.9 (Grade~A,
Excellent) validates the overall approach, and survey feedback confirms perceived
novelty and practical value, particularly for group trips and scenarios requiring
careful coordination.

\section{Critical Reflection and Lessons Learned}
Several lessons emerged during development and evaluation.

First, Git concepts proved powerful but cognitively expensive. During Evaluation
Day, even participants who understood the UI hesitated when encountering terms
like branch and merge. The Easy mode toggle proved essential, but
discoverability and mental model support remain ongoing work.

Second, structured merge turned out to be highly domain-specific. Early
iterations used \texttt{JSON.stringify}-based comparison, which caused subtle
bugs when key ordering differed between branches. Switching to deep structural
equality, stable IDs, and explicit deletion semantics resolved these issues but
required substantial reworking of the merge engine.

Third, constraint scheduling requires explainability. Feedback from testing
showed that users trusted the auto-scheduler more when adjustments (e.g.,
opening-hours nudges) were visibly annotated with reasons rather than applied
silently.

Finally, external dependency management matters more than anticipated. During development, OpenRouteService rate limits and occasional Google Directions failures highlighted the need for Haversine-based fallback strategies to keep the planner usable under degraded conditions.

\section{Limitations}
Several limitations remain. The accessibility of interactive components is the
most significant: maps and drag-based timelines remain challenging for
keyboard-only and screen-reader users. Merge granularity presents a trade-off ---
while field-level conflicts are supported for selected fields, a fully general
per-field resolver could overwhelm users rather than help them. GiTrip supports
asynchronous collaboration via branches and merges but does not offer
simultaneous live editing (CRDT/OT), limiting its suitability for real-time
co-planning sessions. Finally, the scheduler prioritises feasibility and
responsiveness over global optimality, meaning that the generated itineraries are
valid but not necessarily optimal with respect to total travel time or cost.

\section{Future Work}
\paragraph{Improved onboarding and guidance.}
Evaluation Day feedback consistently identified Git terminology as confusing. Future iterations should add in-product tutorials for branching and merging, contextual tooltips explaining versioning in plain language, and clearer ``Save'' semantics bridging the mental-model gap.

\paragraph{Accessibility enhancements.}
The current timeline and map editors rely on mouse-based drag interactions, which
remain inaccessible to keyboard-only and screen-reader users. Future work should
provide keyboard-operable alternatives for timeline edits (e.g.\ arrow keys with
step increments), text-based route editing as a fallback, and stronger Accessible Rich Internet Applications (ARIA)
landmark and live-region support throughout the application.

\paragraph{Real-time collaboration.}
A natural extension is to explore CRDT-based shared plan editing, enabling multiple users to edit the same itinerary simultaneously while preserving a mergeable history model for offline and divergent edits.

\paragraph{Richer merge resolution.}
The merge engine currently detects conflicts on a fixed set of fields (times,
name, coordinates, notes). Future versions could expand conflict detection to
additional metadata, include a live merged-plan preview within the resolution
UI, and support partial merges where users accept some changes and defer others.

\paragraph{Routing extensions.}
The current routing integration supports driving, walking, cycling, and transit
via ORS and Google Directions. Useful extensions include explicit support for
ferries and flights as route legs, integration with General Transit Feed
Specification (GTFS) data where
applicable, and improved offline routing approximations for areas with limited
API coverage.

\paragraph{Optional ML/AI extensions.}
ML/AI features were excluded to preserve determinism and explainability (Section~\ref{sec:ai_scope}) but represent a promising future direction: learning user preferences for scheduling defaults, predicting dwell times from historical data, and context-based destination recommendations. Any such extensions should
be implemented cautiously to maintain the explainability and deterministic
feasibility guarantees that the current system provides.

\chapter*{Author's Assessment of the Project}
\addcontentsline{toc}{chapter}{Author's Assessment of the Project}
% ======================================================================

\textbf{What is the (technical) contribution of this project?} GiTrip contributes
a practical system that applies Git-style branching, committing, and three-way
merging to structured itinerary data, integrated with constraint-aware scheduling
and interactive timeline/map editing.

\textbf{Why is this contribution relevant to Computer Science?} The project
connects software configuration management concepts (history, merging, conflict
resolution) with human-centred planning, demonstrating how domain semantics
change the design of storage, algorithms, and user interfaces.

\textbf{How can others make use of the work?} Developers can reuse the
snapshot-based repository model, merge engine patterns for structured JSON, and
the planner's constraint-handling approach in other collaborative planning
domains.

\textbf{Why should this project be considered an achievement?} It delivers an
end-to-end deployed application combining routing integration, a scheduler, and a
merge UI; evaluation shows users can complete core tasks successfully with limited
guidance.

\textbf{What are the limitations?} The main limitations are accessibility for interactive controls, dependence on external routing providers, and the absence of real-time collaboration and AI-based recommendation within the MVP scope.
