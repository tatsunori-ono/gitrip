\section{Introduction}

This Project Management Report summarises how GiTrip was planned, organised and
executed. It complements the Final Report by focusing on process rather than
technical design. Evidence is drawn from the Project Specification, the
maintained schedule, supervision records, and the project's version control
history (over 200 commits on GitHub). The report addresses six key questions:
development methodology, time management, stakeholder relationships, risk
management, unforeseen issues, and legal/ethical/professional considerations.

\section{Development Methodology}
\label{sec:methodology}
The Project Specification defined a milestone-driven plan to match fixed university deadlines. The project followed an \emph{iterative waterfall} approach: milestones remained stable, while work within each milestone was refined through short feedback cycles. Supervisor meetings were held regularly throughout both terms (Term~1 Weeks 1, 2, 4, 5, 7, 9 and Term~2 Weeks 1, 2, 4, 6, 8, 10), with additional sessions scheduled near submission deadlines to review drafts and incorporate feedback. This balanced predictable progress reporting for supervision with early discovery of integration risks (routing providers, geocoding, and interactive editing). Early prototypes validated feasibility of snapshot storage, server-rendered UI, and routing integration before committing to the full architecture. Must-haves were implemented first to reach an end-to-end MVP; Should-haves and hardening tasks followed. For example, in Term~1 Week~7, API quota exhaustion was identified during a supervisor meeting, prompting immediate caching and fallback implementation. The merge engine required three major rewrites as edge cases emerged during testing, which validated the decision to iterate early on high-risk components. This reduced technical risk by identifying integration issues early and ensuring the system remained demonstrable throughout Term~1.

Development did not begin immediately. Early work (before substantial
implementation) consisted of competitor analysis, a focused literature review
(version control and three-way merge concepts, scheduling heuristics, and
usability heuristics), and a preliminary survey in November 2025. These
activities produced the Project Specification, MoSCoW objectives, and an
evaluation plan, reducing the risk of building features that did not address
real user needs.

\section{Time Management and Work Tracking}
Time management combined a Gantt-style plan with weekly goal-setting and review.
The plan was organised around Term~1 delivery of the end-to-end minimum viable
product (MVP) and Term~2 hardening (tests, robustness, deployment, and evaluation
readiness). Weekly work was prioritised using MoSCoW (Must/Should/Could/Won't),
protecting Must-have items first.

Progress was monitored using three artefacts:
\begin{itemize}
  \item \textbf{Supervision notes:} each meeting used a prepared agenda recorded
        in Obsidian with a structure inspired by the Cornell note-taking
        system~\cite{ref_cornell}, splitting the note into three sections:
        ``Updates from Last Meeting and General Notes'', ``Questions to Ask and
        Answers'', and ``To-dos and Summary''. Whenever questions arose between
        meetings, they were recorded in the ``Questions to Ask'' section of the
        next meeting file. Notes were additionally backed up via an automated
        GitHub plugin sync (every 15 minutes) to reduce the risk of data loss.
  \item \textbf{Source control history:} all work was committed to GitHub for
        backup and traceability; by completion the repository contained well over
        200 commits and feature branches for experimentation.
  \item \textbf{Living schedule:} the original Gantt chart from the Project
        Specification was reviewed weekly. When a three-day delay emerged due to
        API quota exhaustion, the timetable was revised to absorb the delay
        through buffer time and tasks were re-prioritised.
\end{itemize}

To sustain momentum, each week ended with an integrated build and a short review

\section{Stakeholder Management}
The key stakeholder was the supervisor, Professor Mike Joy. As described in
Section~\ref{sec:methodology} (Development Methodology), 12 meetings were held
across both terms. Each meeting followed a structured process: before the
meeting, progress updates and questions were recorded in an Obsidian note under
``Updates from Last Meeting'' and ``Questions to Ask'' sections. During the
meeting, the supervisor's responses were noted beside each item. After the
meeting, action items were summarised in a ``To-dos'' section and converted
into the next week's tasks. This structured approach reduced rework and ensured
progress stayed on track by systematically organising outstanding tasks.

\section{Unforeseen Issues and How They Were Handled}
The project remained broadly aligned to the Project Specification and MoSCoW
priorities, but several unforeseen issues required adaptation and divergence from the original Project Specification.

\textbf{API quota exhaustion.} OpenRouteService rate limits were hit during
intensive testing, causing the planner to return errors. This was addressed by
implementing a TTL cache, bounded concurrency (maximum four simultaneous
requests), and a Haversine-based fallback so the planner degrades gracefully
rather than failing entirely. The fix required schedule adjustment but was
absorbed by buffer time.

\textbf{Merge edge cases.} Early merge testing revealed that
\texttt{JSON.stringify}-based comparison produced false positives when key
ordering differed between branches. This required a substantial rework of the
merge engine to use deep structural equality and stable stop identifiers,
completed before Evaluation Day and backed by 16 automated tests.

\textbf{Place search typo sensitivity.} Nominatim search was sensitive to minor
spelling variations (e.g.\ ``stick n sushi'' for ``Sticks'n'Sushi'') and often
returned no results for near-miss queries, because Nominatim performs exact
index matching. This was resolved by adding Photon as a fuzzy-search fallback,
called only when Nominatim returns zero results. Photon uses the same
OpenStreetMap data but with Elasticsearch-based typo-tolerant indexing.

\textbf{Constraint-aware interactive editing.} Timeline drag-and-drop
interactions could easily produce invalid plans (overlaps, insufficient travel
time, or violations of daily active hours). A two-tier validation strategy was
adopted: the UI enforces minimal constraints with immediate visual feedback
during edits, while server-side validation performs full feasibility checks on
commit.

\textbf{Increased operational work.} Deployment hardening (security headers,
CSP, HTTPS enforcement, rate limiting) and public deployment on Fly.io were not
originally scoped but proved necessary to support reliable demonstrations and
remote testing. The Project Specification boundary excluding generative AI
remained unchanged to avoid scope creep.

\section{Legal, Ethical, Social and Professional Issues}

\subsection{Ethical Consent and Human Participation}
The project included two human-participation activities: a preliminary survey
(November 2025, 30 responses) and user acceptance testing on Evaluation Day
(February 2026, 13 responses). Both involved low-risk tasks: answering questions
about planning habits and completing a guided trial of the software,
respectively. Participants were students from Warwick DCS and the author's
personal network. Responses were collected anonymously via Microsoft Forms, and
results are reported in aggregate statistics. Formal ethical approval was not
required for this scope, but the study followed good practice: minimal data
collection, no sensitive questions, and the ability for participants to withdraw
at any time. If GiTrip were to become a commercial product, a more robust study
with formal consent procedures (information sheet, explicit consent, and clearer
withdrawal mechanisms) would be required.

\subsection{Data Protection and Privacy}
GiTrip minimises personal data collection and avoids sensitive categories. For
user accounts, only essential identifiers (email and username) are stored, along
with bcrypt-hashed passwords. Session cookies are HttpOnly and SameSite=Lax.
Access control is enforced via repository visibility settings and collaborator
roles. Users can request the export or deletion of their data. The design aligns
with General Data Protection Regulation (GDPR) principles \cite{ref_gdpr}.

\subsection{Third-Party Services and Legal Compliance}
GiTrip integrates with several third-party services: OpenStreetMap (map tiles),
Nominatim and Photon (geocoding), OpenRouteService (routing matrices), Google
Directions API (transit legs), GNews API (travel alerts), and Open-Meteo
(weather forecasts). Usage policies for each service were respected through
caching, rate limiting, and bounded concurrency. Required attribution for map
data is displayed in the UI \cite{ref_osm_tiles}. API keys are stored as
environment variables and never exposed to clients.

\subsection{Accessibility}
Accessibility considerations followed WCAG~2.2 guidance \cite{ref_wcag22}. The
application provides font-size adjustment, a high-contrast mode, and a
read-aloud mode for key content, as shown in Figure~\ref{fig:accessibility}.
Labelled form controls and semantic HTML are used throughout. Remaining
limitations (interactive map and timeline drag controls are not fully
keyboard-accessible) are documented in the Final Report rather than concealed.

\subsection{Professional Conduct}
Professional behaviour was guided by the BCS Code of Conduct \cite{ref_bcs}.
This included honest reporting of evaluation results (including limitations and
negative findings), proper attribution of third-party tools and data sources,
and transparent documentation of scope boundaries and known issues.

\section{Conclusion}
GiTrip was delivered through evidence-based planning, milestone-driven execution,
and iterative refinement under regular supervision. Risks were identified early
and mitigated through fallbacks, testing, and careful scope control. The project
was conducted professionally, with explicit attention to ethics, privacy, legal
compliance for third-party services, and accessibility expectations.
